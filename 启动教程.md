# API Test Agent 启动教程 🧪

> **给定一个 OpenAPI 文档 URL，Agent 自动完成解析→分析→生成用例→执行→验证报告全流程。**
> 基于 LangGraph + DeepSeek，7 节点 Pipeline（含 ReAct 反思循环），SSE 实时推送，3 种报告格式（HTML/Markdown/JSON）。
> **前端双版本**: React + TypeScript (主) / Streamlit (兼容)。

---

## 准备清单（5 分钟）

- [ ] Python 3.10+：`python --version`
- [ ] Node.js 18+：`node --version`（React 前端需要）
- [ ] 安装 Python 依赖：`pip install -r requirements.txt`
- [ ] 安装前端依赖：`cd frontend-react && npm install`
- [ ] 配置 API Key：`cp .env.example .env` → 编辑填入 `DEEPSEEK_API_KEY`
- [ ] 目标 API 就绪（本地 `http://localhost:8000` 或任意 OpenAPI URL）

> **不需要 Docker**（Qdrant 非必需，仅用于高级 RAG 功能）：`python setup_and_test.py --skip-docker` 可直接跑。

## Agent Pipeline 流程

```
Spec URL
   │
   ▼
[1. Parse]          ← 抓取并解析 OpenAPI 文档
   │
   ▼
[2. Analyze]        ← LLM 分析端点风险等级，确定测试优先级
   │
   ▼
[3. Generate]       ← LLM 生成测试用例（normal/boundary/error/security）
   │
   ▼
[4. Execute]        ← 真实 HTTP 请求发送（令牌桶限速 + 熔断器保护）
   │
   ▼
[5. Validate]       ← Schema 验证 + LLM-as-Judge 语义分析
   │
   ▼
[6. Reflect]        ← ReAct 反思：LLM 分析失败原因 → 修正用例
   │                   ┌── 有修正项且未达上限 → 回到 [4. Execute]
   ▼                   │
  END  ←───────────────┘  无修正项或已达上限 → 结束
```

6 类自动生成的测试用例（58+ 安全 Payload）：
- **normal**: 合法参数，期望 2xx
- **boundary**: 空 body / 超大 body / unicode / 极端值 / 零值
- **error**: 错误 HTTP 方法 / 错误 Content-Type / 畸形 JSON
- **security**: SQL 注入(16) / XSS(13) / 命令注入(8) / 路径遍历(6) / SSRF(7) / JWT 攻击(6) / CSRF
- **ReAct 自我纠错**: LLM 反思失败 → 修正测试 → 回环重试（最多 2 轮）

## 常见问题

**Q: DeepSeek API 连不上 / SSL 错误？**
> 系统代理干扰。已修复——LLM 客户端使用 `trust_env=False` 绕过代理直连。

**Q: 测试通过率低？**
> 正常。目标 API 的端点需要特定参数或认证才能 2xx，Agent 会标记为 boundary/error 类测试。

**Q: 不想装 Docker / Qdrant？**
> 不影响核心功能。Qdrant 仅用于高级 RAG 检索，跳过 Docker 检查：`python setup_and_test.py --skip-docker`

**Q: Docker 启动失败 / Qdrant unhealthy？**
> 官方 Qdrant 镜像不含 curl，原 healthcheck 用 curl 会报 unhealthy。已在 docker-compose.yml 中修复为 bash TCP 检测（`echo > /dev/tcp/localhost/6333`）。如果仍有问题，确认端口 16333/8002/8501 未被占用。

**Q: 覆盖率只有 25%？**
> 正常。单元测试仅覆盖纯逻辑 `src/tools/` 模块，其余模块（agent 节点、LLM、存储）依赖外部服务，由 E2E 测试覆盖。

**Q: 报告中文乱码？**
> 使用 UTF-8 终端（Windows Terminal / Git Bash），不要用 cmd。HTML 报告强制 UTF-8 编码，不受终端影响。

**Q: 历史测试报告下载 404？** 🆕
> 已修复（2026-07-04）。旧测试运行在数据库中存储了 Windows 主机路径，Docker 容器内无法识别。修复后通过提取文件名 + 运行时 reports 目录重新解析，兼容 Windows/Linux 跨平台路径。`src/main.py` 中 `get_report` 端点已更新。

**Q: 代理（7897 端口）导致 Docker 构建失败？**
> Docker Linux VM 无法访问 Windows 宿主机代理。构建时关闭系统代理，或在 Docker Desktop 设置中移除代理配置。

## 可观测性（LangFuse + CostTracker）

Agent 内置完整的可观测性体系，涵盖调用链追踪和成本统计，**所有功能均为惰性连接——不可用时不影响核心测试流程**。

### 11.1 LangFuse 全链路 Tracing

**自托管 LangFuse 服务**（默认 `http://localhost:3000`），实时采集每次 Agent 运行的完整调用链：

| 采集维度 | 说明 |
|----------|------|
| **Trace（运行级）** | 每次 `run_agent()` 创建一条 Trace，记录 spec_url、thread_id、总耗时 |
| **Span（节点级）** | 6 个 Pipeline 节点（parse / analyze / generate / execute / validate / reflect）各自创建 Span，记录**输入 → 输出 → 耗时** |
| **Generation（LLM 调用）** | 每次 LLM API 调用的 model、token 用量（input/output/total）、耗时 |
| **Event（工具调用）** | 每次工具执行（spec_parser / http_client / schema_validator / report_gen 等），输入输出截断 500 字符 |
| **Event（错误）** | 运行时异常自动记录为 ERROR 级别事件 |

**启动 LangFuse（可选）：**

```bash
# 方式 1：Docker 自托管（推荐）
docker run -d --name langfuse \
  -p 3000:3000 \
  -e DATABASE_URL=postgresql://... \
  -e NEXTAUTH_SECRET=mysecret \
  -e NEXTAUTH_URL=http://localhost:3000 \
  langfuse/langfuse:latest

# 方式 2：LangFuse Cloud（无需自托管）
# 注册 https://cloud.langfuse.com 获取 API Key
```

**配置 .env：**

```bash
# LangFuse 可观测性（不填则自动禁用，不影响核心功能）
LANGFUSE_HOST=http://localhost:3000
LANGFUSE_PUBLIC_KEY=pk-lf-xxxxxxxx
LANGFUSE_SECRET_KEY=sk-lf-xxxxxxxx
```

**查看 Tracing 数据：**

1. 浏览器打开 `http://localhost:3000`（自托管）或 `https://cloud.langfuse.com`
2. 左侧菜单 → **Traces** → 看到 `api-test:{run_id}` 命名的 Trace 列表
3. 点击任意 Trace → 展开查看每个节点的 Span 瀑布图
4. 点击 LLM Generation → 查看具体的 Token 用量和耗时

```
LangFuse Dashboard
  └─ Traces
       └─ api-test:abc123...           ← 一次运行的完整 Trace
            ├─ span: parse             ← 解析节点（输入 URL，输出接口列表）
            ├─ span: analyze           ← 分析节点（LLM 分析风险，输出优先级）
            ├─ span: generate          ← 生成节点（LLM 生成测试用例）
            ├─ span: execute           ← 执行节点（HTTP 请求，输出响应）
            ├─ span: validate          ← 验证节点（LLM-as-Judge 语义判断）
            └─ span: reflect           ← 反思节点（LLM 分析失败，生成修正）
                 └─ span: execute      ← 修正后重试（最多 2 轮）
```

### 11.2 CostTracker 多模型费用统计

**自动记录每次 LLM 调用的 Token 用量和费用**，支持多模型混合计价：

| 模型 | 输入价格 (每 1M tokens) | 输出价格 (每 1M tokens) |
|------|------------------------|------------------------|
| `deepseek-chat` | $0.14 | $0.28 |
| `deepseek-reasoner` | $0.55 | $2.19 |
| `gpt-4o` | $2.50 | $10.00 |
| `gpt-4o-mini` | $0.15 | $0.60 |

**统计指标**（每次运行自动汇总）：

| 指标 | 说明 | 获取方式 |
|------|------|----------|
| `calls` | LLM API 调用次数 | `cost_tracker.call_count` |
| `tokens.input` | 输入 Token 总量 | `cost_tracker.total_tokens["input"]` |
| `tokens.output` | 输出 Token 总量 | `cost_tracker.total_tokens["output"]` |
| `tokens.total` | Input + Output 合计 | `cost_tracker.total_tokens["total"]` |
| `cost_usd` | 美元费用（4 位小数） | `cost_tracker.total_cost_usd` |
| `cost_cny` | 人民币费用（汇率 7.2） | `cost_tracker.total_cost_cny` |
| `models_used` | 本次使用的模型列表 | `cost_tracker.summary()["models_used"]` |
| `duration_s` | 运行总耗时（秒） | `cost_tracker.summary()["duration_s"]` |

**代码中打印费用：**

```python
# 在 run_agent() 之后
from src.observability import cost_tracker

summary = cost_tracker.summary()
print(f"LLM 调用: {summary['calls']} 次")
print(f"Token: {summary['tokens']['total']} (in: {summary['tokens']['input']}, out: {summary['tokens']['output']})")
print(f"费用: ${summary['cost_usd']} ≈ ¥{summary['cost_cny']}")
print(f"模型: {', '.join(summary['models_used'])}")
print(f"耗时: {summary['duration_s']}s")
```

**推理 Token 统计**：使用 `deepseek-reasoner` 模型时，`tokens.output` 包含推理链（reasoning trace）的 Token 消耗，可在 LangFuse Traces 中展开 Generation 查看 reasoning_content 的 Token 占比。

### 11.3 成本优化建议

**Agent 运行完成后自动分析高消耗端点**，在终端和报告中给出提示：

```
=== 成本优化建议 ===
本次测试消耗 $0.0234 (≈¥0.17)
- 1 次 analyze（API 分析）: $0.0002
- 3 次 generate（测试用例生成）: $0.0180 ← 最高
- 1 次 reflect（反思修正）: $0.0032
- 1 次 validate（LLM 裁决）: $0.0020
建议:
  1. 减少 MAX_TEST_CASES_PER_ENDPOINT 可显著降低 generate 成本
  2. 设置 MAX_REFLECT_ITERATIONS=1 可减少反思环节费用
```

**省成本配置**（在 `.env` 中设置）：

```bash
# 每个端点最多生成的测试用例数（默认 8，可降至 4 省一半费用）
MAX_TEST_CASES_PER_ENDPOINT=4

# ReAct 最大反思迭代次数（默认 2，可降至 1）
MAX_REFLECT_ITERATIONS=1

# 仅对高风险端点启用 LLM 生成（中/低风险用规则模板，免费）
LLM_GENERATE_HIGH_RISK_ONLY=true
```

### 11.4 惰性连接设计

**所有可观测性组件均为惰性连接**——未配置时静默降级，不影响 Agent 核心功能：

| 组件 | 未配置时的行为 |
|------|---------------|
| LangFuse | `LANGFUSE_PUBLIC_KEY` 或 `LANGFUSE_SECRET_KEY` 为空 → `TraceManager.available = False`，所有 `start_trace()` / `end_trace()` 等方法直接 return |
| CostTracker | 不受 LangFuse 影响，始终独立运行——即使 LangFuse 不可用，费用统计仍然生效 |
| LangFuse 服务宕机 | 模块加载时 `try/except` 捕获异常，设为不可用——Agent 正常启动，Tracing 静默跳过 |

**验证逻辑**：`src/observability/tracing.py` 中 `TraceManager` 的每个方法第一行都是 `if not self.available: return`，确保 LangFuse 不可用时零开销。

### 12.4 Golden Dataset 回归测试（5 个场景）

**文件**：`tests/golden_dataset.json`  
**用途**：每次改动后验证 Agent 核心能力是否退化

| ID | 场景 | 验证目标 |
|----|------|----------|
| `golden-001` | Mock Buggy API（含 5 个已知 Bug） | Agent 应检出至少 3 个 Bug，通过率 5-40% |
| `golden-002` | Healthy API（无 Bug） | Agent 应给出高通过率（≥75%），避免误报 |
| `golden-003` | Swagger Petstore（公共 API） | 标准 OpenAPI 3.0 解析能力，≥15 端点 |
| `golden-004` | `$ref` Resolution | Schema 引用链正确展开 |
| `golden-005` | Security Injection | SQLi 注入检测，Agent 应标记为安全风险 |

**运行 Golden Dataset 评测**：

```bash
cd api-test-agent
python -c "
from src.evaluation.golden_dataset import run_golden_evaluation
results = run_golden_evaluation()
for r in results:
    print(f'{r[\"id\"]}: pass_rate={r[\"pass_rate\"]:.0f}%  bugs_found={r[\"bugs_found\"]}')
"
```

### 12.5 Mock API E2E 测试

**文件**：`demo/mock_api.py`  
**说明**：一个**故意插入 5 个 Bug** 的演示 API，用于验证 Agent 的 Bug 检测能力：

| Bug | 端点 | 表现 |
|-----|------|------|
| 未处理 limit=0 | `GET /api/users?limit=0` | 返回 500 |
| 不校验 body | `POST /api/users` | 空 body 也接受 |
| 错误 Content-Type | `GET /api/users/999` | 返回 HTML 而非 JSON |
| 无认证检查 | `DELETE /api/users/{id}` | 任意删除 |
| SQL 注入未转义 | `POST /api/users` | 注入 payload 存入数据库 |

**运行 Mock API + E2E 测试**：

```bash
# 终端 1：启动演示 API
cd api-test-agent
python demo/mock_api.py
# 默认监听 http://localhost:8001

# 终端 2：运行 Agent 测试
python -c "
import asyncio
from src.agent.graph import run_agent
state = asyncio.run(run_agent('http://localhost:8001/openapi.json'))
results = state['execution_results']
passed = sum(1 for r in results if r['passed'])
bugs = [r for r in results if not r['passed'] and r.get('severity') == 'high']
print(f'Total: {len(results)}  Passed: {passed}  High-severity bugs found: {len(bugs)}')
for b in bugs:
    print(f'  - {b[\"endpoint\"]}: {b[\"description\"][:80]}')
"
```

### 12.6 一键验证脚本

```bash
cd api-test-agent
python setup_and_test.py
```

**自动执行 7 步**：Python 版本 → .env 配置 → pip 依赖 → Docker 状态 → 单元测试 → 综合测试 → Mock E2E，最终打印通过/失败汇总。

**常用参数**：

| 参数 | 用途 |
|------|------|
| `--check-only` | 仅检查环境，不运行测试 |
| `--skip-docker` | 跳过 Docker 检查（无 Docker 环境时使用） |
| `--skip-unit` | 跳过单元测试 |
| `--skip-comprehensive` | 跳过综合测试 |
| `--skip-e2e` | 跳过 E2E 测试 |
| `--test-url <URL>` | 使用指定 OpenAPI URL 进行 E2E 测试 |

**示例**：

```bash
# 仅检查环境
python setup_and_test.py --check-only

# 跳过 Docker + 使用外部 API
python setup_and_test.py --skip-docker --test-url https://petstore3.swagger.io/api/v3/openapi.json

# 快速验证（仅单元测试）
python setup_and_test.py --skip-comprehensive --skip-e2e
```
